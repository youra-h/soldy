У меня есть проект ui компоненты. Это компоненты под разные фреймворки, чтобы их можно было использовать я разделил проект на несколько слоев:
- core главный слой, который осдержит бизнес логику, у него объявлены свойства и события компонента
- plugins, слой который расширяет core, в нем находится пользовательская логика, здесь описано как будет вести себя компонент при взаимодействии с пользователем, скролинг, disabled, клики и т.д.
- ui/vue, ui/react, ui/angular, ui/svetle и т.д., здесь лишь представление компонента, здесь нет логики, здесь все что выдает core, plugins должно быть отображено одинаково для всех компонентов
- слой setup, provide, здесь я пытаюсь наладить взаимодействие core, plugins, ui. Это слой в котором ме нужна помощь. Необходимо доработать эти слои. Я прикрепил код setup.js в котором есть описание компонентов, как я это вижу. Необходимо добавить слой абстракций в provide и доработать компиляцию компонентов в setup.
И так что мы видим в setup:
- contributions, здесь описание доступных свойств, событие и тригеров у компонента
Например:
    export const ComponentViewContribution: IContribution = {
	props: [
		{ name: 'tag', triggers: ['change:tag'] },
		{ name: 'classes', protected: true, triggers: ['change:classes'] },
	],
	events: ['ready'],
}
У ComponentView есть два своих свойства, tag, classes. Свойство tag публичное свойство, его можно менять пользователю, оно может быть доступно через
<component-view tag="div"></component-view>
свойство classes защищенное свойство, оно не доступно пользователю для изменений, но его можно получить с помощью события change:classes
<component-view @change:tag="" @change:classes="" @ready=""></component-view>
и список events, это список событий которые так же доступны пользователю.
Все тригеры и события для данного компонента находятся в core/TComponentView
triggers в данном случае сообщают о том, что свойство изменилось. Дело в том, что например фреймворк Vue используя Proxy не может знать, когда изменилось внутренее свойство, особенно если свойство является объектом, например свойства visible, rendered, поэтому я использую тригеры и придумал функцию userRefs в ui/vue чтобы сообщать vue, что свойство изменилось и он смог его корректно отрендерить
export function useRefs<T>(
	events: IEventSource,
	getter: () => T,
	triggers: string[],
): Ref<T> {
	let _trigger: () => void

	const ref = customRef<T>((track, trigger) => {
		_trigger = trigger

		return {
			get() {
				track()
				return getter()
			},
			set() {},
		}
	})

	for (const event of triggers) {
		events.on(event, _trigger!)
	}

	onUnmounted(() => {
		for (const event of triggers) {
			events.off(event, _trigger!)
		}
	})

	return ref
}


И так описание доступных свойст,событий для компонента готово, их надо будет вывести в ui.
- помимо компонентов у нас есть такое же описание плагинов, плагины это лишь расширения компонентов, поэтому они описываются точно так же, как и компоненты.
Например:
export const ElementContribution: IContribution = {
	events: ['ready', 'removed'],
}
Здесь как видно, нет доступных свойств для пользователя, но есть доступные события. Если события описаны, значит их необходимо вывести для компонента в котором используется данный плагин  и соотвествено быть доступным для пользователя. Но есть важно уточнение. Можно заметить что события ComponentViewContribution.ready и ElementContribution.ready повторяются, значит мы не можем использовать один и тот же нейминг в итоге для компонента. В описании мы оставляем и там и там ready, но в выводе для пользователя должен быть namespace:
<component-view @ready="" @element:ready=""></component-view>
Namespace element можно взять, когда плагин создается как descriptor, через TElementPlugin.key. Само понятие namespace хорошо описывает именно то с чем мы рабоатет, если мы используем событие @element:ready то мы знаем что работает с плагином который выдаст нам html element.
Namespace кроме плагинов больше нигде не нужен, так как плагины являются расшриением core, все сущности core выводятся как есть вописании, а вот сущности плагинов должны выводиться вместе с namespace!

- descriptiors / definition
Например:
export const ComponentViewDescriptor = defineComponent({
	ctor: TComponentView,

	extends: ComponentDescriptor,

	contribution: ComponentViewContribution,

	plugins: [
		definePlugin({
			ctor: TElementPlugin,
			contribution: ElementContribution,
			provider: TElementPluginAccessorProvider,
		}),
		definePlugin({
			ctor: TInstancePlugin,
		}),
	],

	provider: TAccessorProvider,
})
Как итог мы получаем полное описание компонента для его компиляции. Мы указываем
- ctor, это класс на основе которого будет создан instance
- extends, мы наследуем родительские свойства, события, плагины
- contribution, мы добавляем свое описание компонента
- plugins, указываем какие плагины будут расширять наш компонент
- provider, провайдер как мне видится, это должен быть класс который умеет взаимодействовать с core, plugins, чтобы протащить все события наверх, чтобы работали все тригеры. Это класс который умеет манипулировать одновременно и классами core и классами plugins, провайдеры можно расширять чтобы дополнять какой то своей логикой, которая не похожа на всех, как пример провайдер TElementPluginAccessorProvider, т.е. здесь может быть какая-то не очевидная логика, мы берем все от родителя и что-то дорабатываем, чтобы все корректно взаимодействовало (возможно для какого-либо события ready например хотим добавить в в параметры еще какой либо meta описание, это как пример). Возможно в архитектуре провайдеры в итоге не нужны будут!!!
Как видно чтобы создать descriptor плагина мы используем
definePlugin({
			ctor: TElementPlugin,
			contribution: ElementContribution,
			provider: TElementPluginAccessorProvider,
		}),
где передаем ctor: TElementPlugin, в свою очередь TElementPlugin имеет key, TElementPlugin.key который будет часть namespace о котором шла речь выше.
Плагин может не иметь описания, у него нет contribution и provider, это говорит о том, что мы передаем плагин и нам не надо передавать свойства и события для пользователя на верх. Этот плагин работает внутри системы. Например:
definePlugin({
			ctor: TInstancePlugin,
		}),


- В setup/adapter есть функция createAdapter, это общая для всех фреймворков функция, она создает instance на основе ctor либо получает instance через своство ctrl. Тоже самое с плагинами. И в конце как мне видиться мы создаем runtime, это мост/тунель между core, plugins < - > ui, где у нас происходит обмен данным через описание свойств, событий и тригеры

В слоей ui/vue/adapter у меня есть несколько функций, чтобы адаптировать provide/setup именно под Vue, все что выдаст setup я пытаюсь адаптировать таким образом, чтобы Vue корректно отрендерил. При этом надо понимать что например React не умеет работать с событиями через двоеточие такими как 'change:visible', поэтому у React будут свои адаптеры которые трансформируют событие в onChangeVisible.

Я описал как выглядит проект, слой core, plugins, это законченные и протестированные слои. Сейчас необходимо доработать слои provide, setup, ui/vue таким образом чтобы по данному описанию это все корректно заработало.
